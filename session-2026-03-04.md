# Session Notes — 2026-03-04

## Topic: `take_risc.w` — Algorithm Analysis

---

**Q: Describe the algorithm that the program in take_risc.w implements.**

`take_risc` is an interactive calculator that computes the **product and quotient of two positive 15-bit integers** — but does so by simulating a gate-level RISC processor one logic gate at a time, via `run_risc()` from `gb_gates`.

### The Core Subroutine: `tri`

The machine runs a single ROM program containing a subroutine called `tri`, which computes the **ternary operation**:

> **result = x · ⌊y/z⌋**

with inputs in registers r1=x, r2=y, r3=z, and the result delivered in r4. It also leaves `(y mod z) − z` in r2 as a side effect, from which the caller recovers the remainder.

The two arithmetic operations are special cases:
- **Multiply m·n**: call `tri(x=n, y=m, z=1)` → n·⌊m/1⌋ = n·m
- **Divide m÷n**: call `tri(x=1, y=m, z=n)` → 1·⌊m/n⌋ = ⌊m/n⌋

### The `tri` Algorithm: Binary Long Division

The algorithm is a classic **binary long division with simultaneous scaling of the multiplier**. It runs in two phases:

**Phase 1 — Doubling (loop `l1`/`l2`)**

Starting from z and x, the loop repeatedly doubles both r3 (z) and r1 (x), counting doublings in r5, until z exceeds the remaining value of y (tracked in r2 as a running subtraction). This finds the largest k such that `2^k · z ≤ y`.

**Phase 2 — Accumulation (loop `l3`/`l4`/`l5`)**

Unwinds bit-by-bit from the most significant down to the least:
1. Try adding the current z back to r2 (`r2 += r3`)
2. If r2 ≥ 0, the current bit of ⌊y/z⌋ is **1**: add the current (scaled) x to the result r4, then subtract z from r2 again
3. Halve both r3 and r1 (arithmetic right shift)
4. Decrement the bit counter r5; loop if ≥ 0

At each step, the algorithm is extracting one bit of the binary representation of ⌊y/z⌋, from MSB to LSB, and accumulating the corresponding scaled multiple of x into r4.

### Overflow Handling

Register r6 is a "sticky overflow" flag. It gets set to 1 if either the left-shift of r1 in phase 1 or the addition into r4 in phase 2 causes overflow. At the end, if r6 is nonzero, the program forces the CPU's overflow status bit by executing `r6 >>>= 4`.

### Summary Flow

```
User inputs m, n
  │
  ├─► risc program: tri(n, m, 1)  →  r4 = m·n, overflow bit
  │     print product
  │
  └─► risc program: tri(1, m, n)  →  r4 = ⌊m/n⌋, r2 = (m mod n)−n
        print quotient and remainder
```

The striking point Knuth is making is that a gate-level netlist for a 16-bit processor — with `1400 + 115·8 = 2320` gates — correctly executes a non-trivial binary division algorithm, and the `run_risc` simulator proves it by propagating values through each gate individually.

**Q: Where does the ternary operation come from?**

It emerges naturally from the structure of the binary long-division algorithm itself.

Standard binary division of y by z works by finding which shifted copies of z fit into y, bit by bit:

```
⌊y/z⌋ = Σ  bit_i · 2^i
```

where bit_i = 1 when z·2^i fits into the remaining dividend.

To compute **x·⌊y/z⌋**, you just replace "accumulate 2^i into the result" with "accumulate x·2^i into the result":

```
x · ⌊y/z⌋ = Σ  bit_i · (x · 2^i)
```

The algorithm doubles both z and x together in phase 1, so at each bit position the correctly-scaled multiple of x is already sitting in r1, ready to be conditionally added to r4. Carrying x along costs nothing extra — the same shift that scales z also scales x.

So the ternary form `x·⌊y/z⌋` isn't a pre-existing operation imported from somewhere; it's what you get when you look at division and ask *"what else can I compute for free while I'm at it?"* The two special cases then fall out:

| Want | Set |
|------|-----|
| m·n (multiply) | x=n, z=1 → n·⌊m/1⌋ |
| ⌊m/n⌋ (divide) | x=1, z=n → 1·⌊m/n⌋ |

It's a classic Knuth move: find the most general form of an operation that collapses into all your needed special cases, then implement just that one thing.

---

**Q: So he invented it?**

Almost certainly not — it's a well-known identity in the domain of integer arithmetic and mixed-radix computation. Knuth documents these kinds of floor/ceiling relationships extensively in TAOCP Vol. 1 (Section 1.2.4) and Vol. 2 (on arithmetic). The operation `x·⌊y/z⌋` appears naturally anywhere you need scaled integer division, such as unit conversion, fixed-point arithmetic, and rational approximation.

What Knuth did here was recognize that a single subroutine implementing the general form makes a compact and elegant demo — one routine, two use cases — which fits the pedagogical purpose of `take_risc`. The invention is the *application* (using it to demonstrate the RISC machine), not the operation itself.

That said, the specific prior source he had in mind is unknown. The relevant TAOCP sections or the SGB book itself might cite one.
